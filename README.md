## О проекте

Компания WebStyle занимается разработкой сайтов под ключ и активно работает с внешними подрядчиками: дизайнерами, копирайтерами и другими исполнителями.

Внутренние сотрудники работают в CRM-системе, однако взаимодействие с подрядчиками происходило через мессенджеры, Excel-таблицы и ручные платежи. Это приводило к потере задач, ошибкам при выплатах и отсутствию прозрачности процессов.

Цель проекта — спроектировать интеграцию CRM WebStyle с сервисом MadeTask для автоматизации:
- работы с внешними исполнителями;
- синхронизации задач и статусов;
- подтверждения выполнения работ;
- автоматических выплат подрядчикам.

## Бизнес-проблема

До внедрения интеграции процессы были разрозненными:

- задачи передавались через Telegram и WhatsApp;
- результаты работ хранились вне CRM;
- выплаты выполнялись вручную через Excel и банковские сервисы;
- отсутствовала связь между задачей, приемкой и оплатой.

Основные последствия:
- потеря задач и правок;
- высокая нагрузка на менеджеров и бухгалтерию;
- риск ошибочных выплат;
- отсутствие сквозной трассируемости;
- сложности масштабирования работы с подрядчиками.

---

## Целевое решение

В рамках проекта спроектирована интеграционная архитектура, в которой:

- CRM WebStyle остается основной системой для пользователей;
- MadeTask используется как внешний сервис учета исполнителей и выплат;
- обмен выполняется через REST API;
- выплаты автоматически инициируются после приемки задачи менеджером.

Проект построен по событийной модели:
изменение статусов задач инициирует интеграционные операции и запуск бизнес-процессов.

---

# Architectural Decisions

В рамках проектирования были приняты следующие архитектурные решения:

- CRM определена как master-system для операционных данных;
- MadeTask используется как внешний payment/execution сервис;
- платежные реквизиты не хранятся в CRM;
- интеграция построена по событийной модели;
- операции создания выплат реализованы идемпотентно;
- синхронизация выполняется через REST API и JSON;
- ошибки API не блокируют работу CRM и обрабатываются через retry-механизмы и журнал интеграции;
- реализована сквозная трассируемость:
  task → execution → acceptance → payout.

# Integration Edge Cases
В проекте дополнительно проработаны сложные интеграционные сценарии:

- отсутствие регистрации исполнителя;
- отсутствие привязанного счета;
- retry-механизмы при временных ошибках API;
- обработка конфликтов статусов (`409 Conflict`);
- защита от повторных выплат через idempotency keys;
- обработка rate limits (`429 Too Many Requests`);
- логирование всех интеграционных операций;
- уведомления пользователей и администраторов при ошибках;
- сценарии временной недоступности MadeTask API;
- механизмы повторной синхронизации статусов.

# Итог проекта

В результате спроектировано интеграционное решение, позволяющее:

- централизовать работу с внешними подрядчиками;
- исключить ручную синхронизацию задач и выплат;
- автоматизировать процесс оплаты после приемки задачи;
- снизить операционную нагрузку на менеджеров и бухгалтерию;
- уменьшить риск потери задач и ошибочных выплат;
- обеспечить сквозную трассируемость бизнес-процесса;
- повысить масштабируемость работы с подрядчиками.

Проект включает полноценный набор артефактов системного аналитика:
от анализа AS IS процессов до API-спецификаций, UML/BPMN-диаграмм, требований и технического задания.
